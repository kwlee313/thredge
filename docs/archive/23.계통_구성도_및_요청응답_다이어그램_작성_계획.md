# 계통 구성도 및 요청/응답 다이어그램 작성 계획

## 배경/문제
- 프로젝트 구성 요소와 상호 연결 구조를 한눈에 파악하기 어렵다.
- Nginx부터 DB까지의 요청 흐름과 응답 흐름을 문서에서 명확히 전달할 필요가 있다.

## 목표
- README.md에 컴포넌트 구성 및 통신 구조를 나타내는 Mermaid 다이어그램을 추가한다.
- 요청/응답 플로우를 Mermaid sequence diagram으로 명확히 표현한다.

## 범위
- 포함: Nginx, 애플리케이션 서버(백엔드), 프론트엔드, 데이터베이스, 외부 연동(있다면) 간 통신 구조
- 포함: 대표적인 요청/응답 시나리오(예: 클라이언트 -> Nginx -> 백엔드 -> DB -> 백엔드 -> 클라이언트)
- 제외: 상세 성능 튜닝, 실제 인프라 변경, 코드 로직 변경

## 현재 동작 정리
- README.md 및 infra/docker-compose 관련 파일에서 현재 구성 요소를 파악한다.
- 백엔드/프론트 폴더 구조와 리버스 프록시 설정을 확인한다.

## 개선 방향
- 다이어그램은 단순화된 상위 수준 구조로 유지한다.
- 구성 요소 명칭은 실제 프로젝트 파일/설정에 맞춘다.
- 요청/응답 흐름은 대표 시나리오 한두 개로 제한한다.

## 단계별 계획
- 1단계: 구성 요소 파악
  - 무엇을 변경: README.md, infra/, docker-compose.yml, nginx 설정 파일 확인
  - 완료 기준: 다이어그램에 들어갈 컴포넌트 목록과 연결 관계 정리
- 2단계: Mermaid 다이어그램 작성
  - 무엇을 변경: README.md에 component diagram 추가
  - 완료 기준: Nginx부터 DB까지의 통신 구조가 한눈에 보이는 다이어그램 삽입
- 3단계: Sequence diagram 작성
  - 무엇을 변경: README.md에 요청/응답 플로우 다이어그램 추가
  - 완료 기준: 대표 요청 흐름이 단계별로 표현됨
- 4단계: 검토 및 반영
  - 무엇을 변경: 다이어그램 표현/명칭 조정
  - 완료 기준: 사용자 검토에서 승인

## 리스크 및 대응
- 실제 구성요소를 잘못 추정할 위험
  - 대응: README.md 및 infra 설정을 근거로 명칭과 연결을 기재
- 다이어그램 과도한 복잡화
  - 대응: 핵심 구성요소만 반영, 세부는 생략

## 검증 계획
- README.md 렌더링 시 Mermaid 다이어그램이 정상 출력되는지 확인
- 주요 구성 요소가 빠지지 않았는지 수동 검토
